# 解題思路與工作方式

> 本次測驗包含兩題:① OCPI 2.2.1 充電漫遊時序圖 ② 會員系統 RESTful API(註冊/開通、兩階段登入、最後登入查詢、部署上雲)。
> 這份文件說明我「怎麼處理、歸納、整理、回應」這些題目。

## 整體策略

**先理解 → 再規劃 → 小步驗證 → 收尾文件化。** 不一上來就寫程式,而是先把題目背後的領域與需求邊界搞清楚再動手。

## 工作方式的五個特徵

| 特徵 | 本次實例 |
|---|---|
| 1. 先建立領域知識再動手 | 畫時序圖前先釐清:OCPI 由誰發布、CPO/eMSP 角色、OCPI vs OCPP、平台串接哪一端 |
| 2. 對照來源、不接受空話 | 時序圖依 OCPI 2.2.1 文件繪製,逐一對照章節(§9 Sessions、§10 CDRs、§13 Commands) |
| 3. 用審查驗證成果 | 主動以「資深 PM 視角」反向 review 時序圖與題目要求的落差並修正 |
| 4. 規劃先行、分階段執行 | API 先定架構與選型,再依里程碑推進:資料模型 → Email → 兩階段登入 → Swagger → 部署 → GitHub |
| 5. 持續驗證與交付品質 | 每階段以測試或實際 API 呼叫驗證;收尾再確認 GitHub 同步、.gitignore 覆蓋、Swagger 欄位有效 |

## 兩題的思路

**第一題(時序圖)**
釐清協定分層(SCSP↔EMSP 非 OCPI、EMSP↔CPO 為 OCPI、CPO↔充電樁為 OCPP)→ 抓出題目要求的 7 個事件 → 對照文件繪製 → 審查驗證 → 確保每條訊息都能對應到文件章節以便口頭說明。重點掌握:**Commands 的非同步雙回應**、**Session/CDR 由 CPO 主動推送**。

**第二題(RESTful API)**
規劃架構 → 選型 → 分階段實作並逐步測試 → 遇阻礙先定位根因再務實轉向 → 部署上線實測 → 文件化。
- 防越權:最後登入查詢的身分一律取自 JWT,不收 userId 參數,結構上即無法查他人。
- 安全:所有金鑰走環境變數 / .gitignore,不進版控。
- 務實排障:Mailjet 新帳號被封 → 改用 Gmail SMTP;Cloud Run 建置權限不足 → 補上 IAM 角色;Swagger 回應欄位空泛 → 改用具型別的回應 DTO。

## 成果

- 時序圖(`docs/ocpi-charging-sequence.md`)、完整可運行的會員系統、12 項自動化測試
- 線上 Demo(GCP Cloud Run + Neon PostgreSQL)、Swagger 文件、GitHub repo
- 過程文件:規劃書、部署紀錄、本解題思路

## 關於工具運用

本次以 AI 輔助完成,我的角色是主導者:負責**拆解需求、技術選型、驗證正確性、取捨決策**;AI 負責加速執行。重點不在「誰寫的」,而在**最終結果正確、可重現、可交接**。
